# AI Session Log — 2026-03-28

## User request summary
- Implement the NFC tap-to-reveal flow with a robust manual fallback
- Must feature-detect Web NFC; iOS Safari and desktop browsers should silently degrade to the manual scan button instead of showing an error
- Permission errors must surface clearly to the user (we got burned in an earlier draft where errors were swallowed)

## Prompt used (paraphrased, not verbatim)
> "Show a React hook `useNfcReader` for Chrome on Android that wraps `NDEFReader`, returns `{ isSupported, isScanning, lastTagId, error, start, stop }`, and degrades gracefully when `NDEFReader` is undefined. iOS Safari and desktop should set `isSupported=false` instead of throwing. When the user denies the permission prompt, the hook must return a non-empty `error` so the UI can render an explanatory banner."

## What the AI returned
- A `useNfcReader.js` hook with the exact return shape we asked for
- Feature detection via `'NDEFReader' in window`
- An async `start()` method that wraps `reader.scan()` in a try/catch

## Changes we accepted vs. rejected
- **Accepted:** the hook signature, feature-detection pattern, scan event listener
- **Rejected:** the first version silently caught `NotAllowedError` and reset `error` to `null` — we replaced it with explicit `setError({ code: 'permission-denied', message: '...' })` so the UI can render a remediation banner
- **Edited:** added a `useEffect` cleanup that calls `reader.stop?.()` on unmount; without it we saw "scan already in progress" errors on hot-reload during development

## Verification performed
- **Pixel 6 (Chrome, NFC supported):** scanned 3 different NTAG215 tags, each returned a stable `lastTagId`; permission revocation through Chrome settings correctly surfaced the error banner
- **iPhone 13 (Safari, NFC unsupported):** `isSupported` returned `false` immediately; the manual scan button rendered without any console error
- **Desktop Chrome (NFC unsupported):** same as iOS — clean degradation to manual flow
- **Edge case:** rapidly tapping start/stop did not leak listeners (verified via `getEventListeners` in DevTools)

## Notes for the team
- Web NFC requires HTTPS in production. GitHub Pages serves HTTPS, so we are fine
- Do not assume NFC support on Android — some older devices have NFC disabled at the OS level even though the API exists
- The fallback "manual scan" button reuses the same `lastTagId` callback, so downstream UI does not need to branch on input source
